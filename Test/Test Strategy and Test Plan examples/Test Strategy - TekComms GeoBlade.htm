1. Test Strategy

1.1. Overview

 

This is the Test Strategy for the Bladed Probe. This document shall be completed and used by the project test team to guide how testing will be managed for this project. The test effort will be prioritized and executed based on the
project priorities as defined in the Project Plan and Requirements Specification. This is a living document that may be refined as the project progresses. The QA Manager, Test Team Lead, Project Manager, and Development Manager shall review and approve the final version of the Test Strategy document, and is subject to change until all PRD requirements have been finalized and scoped in to the release.

1.1.1. Audience

This document is intended for the following audiences:

  • Feature QA Engineers
  • System Test QA Engineers
  • Development primes
  • Project / Development Managers

Testing of the bladed probe will include multiple phases, encompassing both scheduled milestones in the overall Product Release cycle, and from milestones in hardware availability schedule.

1.1.2. Definitions

  • Scenario - Detailed description (specific instance) of a use case, including rules, exceptions, boundaries, limits, etc.
  • Test Coverage - Describes how much of a system has been tested
  • Test Plans - Describes how a feature or function shall be tested
  • Testcase - A specific set of test data along with expected results for a particular test objective, including specific steps for execution and analyzing of results
  • Test Strategy - This document. Describes the scope, approach, resources and schedule for the testing activities of the project. This includes defining what will be tested, who will perform testing, how testing will be managed, and the associated risks and contingencies.

 

 

1.2. Scope and Limitations

 

At this time, Tektronix software release 14.1 is targeted for this product and thus this test strategy document. Business requirements could result in the testing of the bladed probe to occur in a subsequent release, pending decisions and schedule still to be determined. Installation will be tested on the different platforms required in the requirements specficiations.

1.2.1. Limitations and Exclusions

This test strategy will lay out the stages as to what functionality can be verified throughout the development cycle.  This test strategy only covers the testing of functionality with the new Bladed Probe. This testing strategy is not intended to cover any new test coverage of existing hardware such as G10, 2U, 14U except as identified with interoperability between the new probe and existing hardware types. these will be covered by the respective feature test teams and system test teams for the release.

1.3. Testing Approach

1.3.1. Timeline

Below is a sketch of the testing phases contrasted against the development lifecycle and hardware lifecycle. Resource personnel requirements are still an estimate at this point, as not all functionality has been locked down yet. Planning will complete once the PP milestone has been met.

Source: QA_Resource_Timeline.pptx

 

Note that the hardware dates identified above are based on the latest component being available for each major milestone (excluding 100G I/O card). In some instances there will be some components available before this milestone, but for simplicity purposes, planning is being done based on prototype, pilot, and pre-production on the system as a whole.

1.3.2. Test Phases

The major phases of testing are:

  • Feature Development Phase (DP to DC)
    • Hardware Validation
    • Automation TC Development
    • Testcase/Testplan creation
    • Feature Testing - Functional testing to be performed by QA engineers assigned to development teams. The QA engineers will begin working on the project at the same time the development team starts (around DP milestone) 
    • Integration Testing - Part of Feature testing, but includes cases where cross functional teams are required
    • System Testing - Testing performed by a system QA engineer, generally including more black box functionality. System engineers will begin working on the project after they free up from the previous release (around FR milestone of the previous release)
  • Hardening Phase (DC to FR)
    • Continued System Testing (hardening) - Retesting of major functional components
    • Robustness Testing
    • Performance testing
    • Providing output to Tekpubs documentation
  • FOA Phase (FR to GA)
    • FOA and soak testing in near customer environments

1.5. Special Test Considerations

 

1.5.1. Hardware Testing

  1. MBC Timing source/propogation - need to have adequate tools to verify timing lag across all cards. Dev loopback ??? Require full shelf for under load timing testing.
  2. Form/fit and low level hardware validation - no concerns except for timing of hardware arrival. Vendor quality testing is key to successful timing.
  3. Fabric backplane testing - need to identify methods to inject traffic and verify crosstalk
  4. Yoda i/o blade - Need traffic generation of multiple 10G input sources for full blade testing. need Mobile Metric tool to be solid. Still TBD

1.5.2. OAM Testing

  1. Panel Breakout - no major considerations. Actual hardware will be required for some statistics testing
  2. Storage Profile - Will require extensive use of disk arrays and time for adequate aging testing.
  3. Topology - Verification of user addition/deletion and dependencies. Only concern will be time to test all configuration possibilities
  4. Bulk Processing - Each area of test will require application robustness and field error handling -
  5. Upgrades - no major considerations
  6. Link Detection - are we doing auto link detection?? Testing here will require access to traffic generation tools.
  7. system boot/recovery - multiple configurations to be tested on system startup

 

1.5.3. Packet Lifecycle Testing

  1. General packet routing testing - Most cases will require multiple interfaces to be injecting traffic and verify routing is properly done. Need to identify what traffic tools are available to do this
  2. distributed processing across multiple I/O blades - how to measure load distribution? do we need KPI's to be added, indicating # of sessions currently being tracked?
  3. Re-balancing and failure scenarios - much testing will be required here to ensure traffic is not lost in event of card failure.
  4. Throttling testing - testability in true capacity environment is preferred over lowering thresholds.

1.5.4. VSS/Tek Integration Testing

  1. Full packet broker functionality - what are the major first deployment usecases?

1.5.5. Storage Testing

  1. iSCSI testing - is this in scope? if so, require allocating time and hardware for testing

1.5.6. Platform Testing

  1. Storage requirements & logging throughput - testing requires generation of heavy log activity. Can a log generation script be built into the Yoda and flow blades for testing?
  2. Storage of Session data and deciphering keys - need HLD completion before being able to solidify testability. Challenges with Redis instances?
  3. Redundancy (MBC) - require sole use of dual MBC cage setup for multiple test scenarios and error injections to force MBC switchover
  4. Redundancy (others - is redundancy for other functions done in Plat? I'm thinking it is not, but need confirmation.)
  5. Packaging - testability of firmware updates
  6. Memory utilization testing - how to track memory usage to determine if leaks exist?

1.5.7. Performance Testing

  1. High traffic generation - risk is that we do not yet have a tool capable of generating high traffic. Availability is TBD.
